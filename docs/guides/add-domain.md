# Adding a new domain to the pipeline

Guide for adding a new test domain (for example `blazeup_superadmin`) to the framework +
CI/CD. Replace `<domain>` with the new domain name (snake_case, for example `blazeup_superadmin`)
and `<DOMAIN>` with that name in UPPERCASE (for example `BLAZEUP_SUPERADMIN`).

> Templates available to copy: `blazeup_admin` (with tests + Excel) and `blazeup_partner`
> (empty skeleton). A consistent naming convention is what lets CI "auto-detect" the new domain
> in most of the steps.

---

## A. Code in the repo

1. `runner/<domain>/` — copy the **3 entry points** from `runner/blazeup_admin/`, changing
   `os.environ.setdefault("BLAZEUP_DOMAIN", "<domain>")` in all 3:
   - `run_test.py` — runs the tests
   - `health.py` — health-check API service (shared engine in `utils/health_check.py`)
   - `swagger_check.py` — swagger drift detector (shared engine in `utils/swagger_check.py`)
   - add `__init__.py` (empty) if it does not already exist.
2. `runner/<domain>/registry.py` — generated by `python utils/sync_registry.py`
   (read from the test plan Excel). Do NOT edit by hand.
3. `tests/<domain>/` — the domain's test cases.
4. `config/<domain>/.env` — for running LOCALLY (gitignored, do not commit):
   ```
   BASE_URL=...
   API_BASE_URL=...
   TEST_EMAIL=...
   TEST_PASSWORD=...
   ```
5. (Optional, if exporting Excel) `docs/<domain>/Partner_Platform_Test_Plan.xlsx`
   and set `REPORT_EXCEL = True` in `runner/<domain>/run_test.py`.

Run a local trial before touching CI:
```bash
python -m runner.<domain>.run_test --mode smoke   # run tests
python -m runner.<domain>.health                  # check whether the service is up/down
python -m runner.<domain>.swagger_check --save     # capture the first Swagger baseline
```

> Note: in `health.py` / `swagger_check.py`, adjust `EXTRA_SERVICES` if this domain
> needs to monitor a service that does not yet have an API client (e.g. `sa-partners-api`).

## B. Workflow `.github/workflows/test.yml` — edit 4 places

### 1) Add an option to the `domain` input
```yaml
      domain:
        type: choice
        options:
          - blazeup_admin
          - blazeup_partner
          - <domain>          # ← add
          - all
```

### 2) Add to the `all` matrix list (job `setup`)
```yaml
            all) DOMAINS='["blazeup_admin","blazeup_partner","<domain>"]' ;;
```

### 3) Add a branch in "Map domain secrets" (job `tests`)
```yaml
            <domain>)
              echo "BASE_URL=${{ secrets.<DOMAIN>_BASE_URL }}"           >> "$GITHUB_ENV"
              echo "API_BASE_URL=${{ secrets.<DOMAIN>_API_BASE_URL }}"   >> "$GITHUB_ENV"
              echo "TEST_EMAIL=${{ secrets.<DOMAIN>_TEST_EMAIL }}"       >> "$GITHUB_ENV"
              echo "TEST_PASSWORD=${{ secrets.<DOMAIN>_TEST_PASSWORD }}" >> "$GITHUB_ENV"
              ;;
```

### 4) Telegram routing (the "Notify via Telegram" step)
Add one env line:
```yaml
          TELEGRAM_CHAT_ID_<DOMAIN>: ${{ secrets.TELEGRAM_CHAT_ID_<DOMAIN> }}
```
And one `case` branch:
```yaml
            <domain>) CHAT_ID="${TELEGRAM_CHAT_ID_<DOMAIN>:-$TELEGRAM_CHAT_ID}" ;;
```

## C. GitHub Secrets to add
Settings → Secrets and variables → Actions → **New repository secret**:

| Secret | Required | Notes |
|---|---|---|
| `<DOMAIN>_BASE_URL` | ✅ | Web URL of the domain |
| `<DOMAIN>_API_BASE_URL` | ✅ | API URL |
| `<DOMAIN>_TEST_EMAIL` | ✅ | Test account |
| `<DOMAIN>_TEST_PASSWORD` | ✅ | Test password |
| `TELEGRAM_CHAT_ID_<DOMAIN>` | ⬜ | Dedicated channel for the team. Leave blank → use the shared channel `TELEGRAM_CHAT_ID` |

Note when entering a secret: do not leave whitespace / blank lines at the end of the value.

## D. Automatic — nothing to do

| Feature | Behavior with a new domain |
|---|---|
| Allure dashboard + trend | Automatically creates `https://<owner>.github.io/<repo>/<domain>/` |
| AI triage (`ai_triage.md`) | Auto-generated when there are test failures (flag `REPORT_AI_TRIAGE`) |
| Excel checkbox / AI triage | Applied uniformly to every domain |
| Telegram | Has a dedicated secret → goes to the dedicated channel; if not → falls back to the shared channel |
| Health-check & Swagger drift | Shared engine (`utils/`) — you only need to copy the thin file in section A.1; it auto-scans services from `api_clients/<domain>/` |

---

## Why do 3 `case` branches have to be edited by hand?
GitHub Actions does **not** allow referencing a secret by a dynamic name at runtime
(there is no `secrets[$domain]`). For security, a secret is only accessible by its **literal
name** `${{ secrets.NAME }}`. Therefore each domain must be declared explicitly in the 3
`case` branches above. This is a platform limitation, not a framework one.

## Quick checklist
- [ ] `runner/<domain>/` (run_test.py + health.py + swagger_check.py + __init__.py + registry.py)
- [ ] `tests/<domain>/`
- [ ] `config/<domain>/.env` (local) + a trial run with `--mode smoke`
- [ ] `python -m runner.<domain>.health` (check the service)
- [ ] `python -m runner.<domain>.swagger_check --save` (Swagger baseline)
- [ ] (optional) `docs/<domain>/...xlsx` + `REPORT_EXCEL = True`
- [ ] Workflow: input option, `all` matrix, Map domain secrets, Telegram route
- [ ] Secrets: `<DOMAIN>_BASE_URL` / `_API_BASE_URL` / `_TEST_EMAIL` / `_TEST_PASSWORD`
- [ ] (optional) `TELEGRAM_CHAT_ID_<DOMAIN>`
- [ ] Run `domain=<domain>` on Actions to confirm green + dashboard + Telegram
